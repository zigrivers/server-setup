# Local Embeddings + Qdrant RAG — Design Spec (F5)

**Status:** APPROVED for build (via /goal). Feature 5 of 6. Implementation home: **server-setup**
(serving tools; Python). The research's caveat governs the design: **bad retrieval hurts more than
no retrieval** — a single answer-less-but-similar distractor can drop accuracy ~25%. So this is
deliberately *conservative*: small top-k, a relevance-score gate, and quality measured (not assumed)
via the F1 eval harness.

## Goal
A local, offline retrieval toolkit: embed a corpus of the user's own docs/code into Qdrant, retrieve
the top few relevant chunks for a query (with a score gate), and build a RAG-augmented prompt — then
**A/B it against no-RAG** so we only adopt it where it measurably helps.

## Non-goals (v1)
- No always-on embedding HTTP server — a small local embedding model loaded by the ingest/query
  scripts is enough for v1 (a persistent endpoint is the documented production path).
- No automatic re-indexing / file watching — ingest is a manual/CLI step.
- No re-ranking model in v1 (score gate + small top-k is the conservative baseline).
- No wiring RAG into every client — v1 provides the toolkit + a RAG eval arm; broad adoption follows
  the measurement.

## Approach
1. **Embeddings** via `mlx-embeddings` with a small local model (e.g. a bge/qwen3 small embedder,
   ~0.3–0.6B). One function `embed(texts) -> vectors`.
2. **Qdrant** (Docker, port 6333) — one collection, cosine distance, `size` = the embedder's dim.
3. **Ingest** `scripts/rag-ingest.py <path...>`: read text/markdown/code files, **chunk**
   (~800 chars, ~100 overlap), embed, upsert into Qdrant with payload `{path, chunk, ord}`.
4. **Retrieve** `retrieve(query, k=4, min_score=0.3)`: embed the query, Qdrant search top-k, **drop
   hits below `min_score`** (the relevance gate — the key conservative control), return the kept
   chunks. k default 4 (research: 3–5).
5. **Augment** `rag_prompt(query, chunks)`: build a prompt that puts the retrieved context with the
   query, **best chunk at an edge** (mitigate "lost in the middle"), and instructs the model to use
   it only if relevant. If no chunk passes the gate → return the query unchanged (no-op, never inject
   noise).
6. **Measure**: a RAG eval arm (the eval harness's `rewriteBaseUrl`-style hook, generalized) or a
   standalone `rag-query.py` that answers with vs without retrieval; A/B via F1. Adopt only if it
   doesn't regress.

## Error handling
- Qdrant down → clear message + exit (ingest/query can't proceed).
- Embedder load failure → clear message.
- Empty corpus / no hits above the gate → query runs **without** context (the safe default), logged.
- Dimension mismatch (changed embedder) → detected at `ensure_collection` (recreate or error clearly).

## Test plan
- Unit: `chunk` (sizes/overlap/edge cases), score-gate (`retrieve` keeps ≥min_score, drops below),
  `rag_prompt` (no chunks → unchanged; chunks → context + edge placement) — no model/Qdrant needed.
- Live: ingest a small corpus (the repo's `docs/` + key scripts), query something answerable from it
  → retrieval returns the right chunk above the gate; an off-topic query → nothing passes the gate
  (no noise injected). Confirm Qdrant collection dim = embedder dim.

## Revisions from multi-model review (incorporated)
- **L2-normalize embeddings** before upsert and query (so Qdrant cosine is well-behaved regardless of
  whether the model normalizes). `embed()` normalizes.
- **Score gate calibrated, not magic.** Default `min_score=0.45` chosen from the *observed*
  bge-small-en-v1.5 distribution on this stack (similar pairs ≈0.74, unrelated ≈0.36 — measured), so
  0.45 separates signal from noise. Configurable; `rag-query.py` prints each hit's score so the gate
  can be tuned per corpus.
- **Block-aware chunking.** `chunk()` splits on blank lines first and packs blocks up to ~800 chars
  (with overlap) rather than slicing mid-line/mid-block — better for code/docs than raw char slicing.
  (Syntax-aware splitting is a documented future step.)
- **Stale-chunk freshness.** Ingest **deletes existing points for a file's `path`** before upserting
  its new chunks, and uses a **stable point id = hash(path + ord)** so re-ingest overwrites instead
  of duplicating.
- **Dedup retrieved chunks** by exact text (overlap can surface near-identical neighbors) so the
  context window isn't wasted on repeats.
- **Defined chunk placement.** Kept chunks are sorted by score **ascending** so the **highest-score
  chunk sits last — adjacent to the query** (mitigates "lost in the middle"); the prompt instructs the
  model to use the context only if relevant.
- **Named collection.** `COLLECTION` env (default `server_setup`) namespaces the corpus so a shared
  Qdrant instance doesn't collide; dim is recorded at creation and a mismatch errors clearly.

## Risks
- **Distractors hurt** — mitigated by small k + the score gate + the no-inject-on-empty rule;
  documented, and the A/B is the real safeguard (we don't adopt RAG where it regresses).
- **Embedder quality / dim** — pick a solid small MTEB embedder; record its dim in the collection.
- **RAM/latency** — a small embedder is cheap; Qdrant is light. The persistent embedding endpoint
  (production) is documented, not built in v1.
- **Scope honesty** — v1 delivers the toolkit + measurement path; it does not auto-enable RAG for
  clients. The eval A/B decides where it's worth it.
