1. Executive Summary

Current target design: use frontier models such as Claude Code and Codex for architecture, planning, documentation, and final judgment. Use local open-source Heretic/uncensored Qwen-family models for implementation, test generation, debugging, and local review loops.

This setup is a hybrid development system. It does not combine two Macs into one shared-memory machine. Instead, it creates a distributed multi-agent workflow: 

Frontier layer:
  Claude Code / Codex
  Roles: senior architect, planner, documentation writer, final reviewer

Machine 1:
  Daily driver / cockpit / control plane
  Runs LangGraph controller, MCP bridge, VS Code/Cursor, Claude Code/Codex
  Runs local Orchestrator model on localhost

Thunderbolt Bridge:
  Direct private Mac-to-Mac network

Machine 2:
  Dedicated inference worker
  Runs Developer, Reviewer, optional Security Reviewer / MTP experimental models

The operating principle is: 

Frontier model writes the contract.
Local agents execute the contract.
Frontier model performs final review.
Human approves merge.

This mirrors the uploaded frontier-to-local handoff plan’s core idea: use frontier model judgment where leverage is highest, then push the high-volume execution phase to local models.

Why this split?

  • Machine 1 stays the cockpit: all developer tools, plan files, worktrees, and frontier agents live where you work.
  • Machine 1 Orchestrator avoids unnecessary Mac 2 load: the Orchestrator is mostly planning/routing, so it fits naturally on the daily driver.
  • Machine 2 focuses on worker models: Developer and Reviewer are where most repeated inference happens.
  • Thunderbolt is fast enough: prompts/responses are tiny compared with model compute. Inference is the bottleneck, not network throughput.

5. BF16 vs 9-bit Strategy

The current precision strategy is intentionally mixed:

RoleRecommended PrecisionReason
Orchestrator BF16 on Machine 1 Planning, routing, judgment, long-range coherence. Quality is worth the memory.
Developer Mixed 9.4-bit on Machine 2 Bulk token generation, patch writing, retries. Throughput and headroom matter.
Reviewer BF16 on Machine 2 Review quality matters more than raw speed.
Experimental extra agents Mixed/high-bit quant or on-demand Avoid memory pressure and keep the worker machine flexible.

Practical rule

Use BF16 where judgment matters.
Use high-bit quantization where throughput and concurrency matter.
With 512 GB unified memory per machine, BF16 fits comfortably for single large models. The real pressure comes from weights + KV cache + long context + multiple simultaneous servers + macOS/app memory.

8. LangGraph Agent System

The local multi-agent controller is implemented in: 

~/ai/agents/three_agent_system/local_agents.py

Agent graph

User task or frontier-generated plan
        │
        ▼
Orchestrator
  - reads task
  - inspects repo tree
  - makes implementation plan
  - defines review checklist
        │
        ▼
Developer
  - writes unified diff patch
  - proposes commands/tests
  - controller applies patch
  - controller runs tests/tools
        │
        ▼
Reviewer
  - reviews diff, tests, logs, task, plan
  - returns JSON approval or required changes
        │
        ├── if rejected and loop count < MAX_REVIEW_LOOPS → Developer again
        │
        └── if approved or max loops reached → final state

Key implementation invariants

  • local_agents.py reads all endpoints and model paths from .env.
  • It validates that model names are not local.
  • Developer output is expected to include <summary>, <patch>, and <commands> blocks.
  • Patches are applied with git apply, with patch -p1 fallback.
  • Commands are blocked if they look destructive unless ALLOW_UNSAFE_TOOLS=1.
  • The controller uses git workspaces and stores memory in .agent_memory.jsonl.

9. Frontier-to-Local Workflow

This setup adopts the plan → local execution → local review → frontier review → human merge model. It is based on the principle that the local model’s success depends heavily on plan quality.

Default serious-work workflow

1. Open Claude Code or Codex in the repo.
2. Ask frontier model to create a concrete plan:
   plans/YYYY-MM-DD-feature-name.md

3. Human reviews and approves the plan.

4. Run local execution:
   local-exec-plan plans/YYYY-MM-DD-feature-name.md --workspace .

5. Local agents execute in a separate git worktree.

6. Inspect:
   git worktree list
   cd ../repo-local-feature-*
   git status
   git diff
   run tests manually

7. Ask Claude Code / Codex for final architecture/docs/security review.

8. Human commits and merges.

Good plan requirements

  • Goal and non-goals
  • Current system summary
  • Proposed architecture
  • Exact files expected to change
  • Step-by-step implementation tasks
  • Function/class signatures when applicable
  • Edge cases
  • Exact test commands
  • Rollback plan
  • Documentation updates
  • Risks and security concerns
A useful test: if a human engineer could implement the plan without asking follow-up questions, it is probably specific enough for the local Developer model.

13. Security Rules

This stack uses uncensored/local models and local tool execution. That makes it powerful and dangerous. Treat it as a junior engineer with shell access: useful, but never fully trusted.

Hard rules

  • Do not expose local model servers to the public internet.
  • Use 127.0.0.1 for Machine 1 Orchestrator.
  • Use 10.10.10.2 for Machine 2 worker endpoints over Thunderbolt.
  • Use Tailscale if remote access is required.
  • Do not let agents operate in your home directory.
  • Prefer git worktrees for all non-trivial execution.
  • Never give agents secrets, private keys, production credentials, or sensitive customer data unless there is a deliberate local-only reason and you understand the risk.
  • Keep ALLOW_UNSAFE_TOOLS=0.
  • Do not allow auto-commit or auto-merge for high-risk changes.
  • Human approval is required for dependency additions, migrations, destructive commands, security-sensitive code, and final merge.

Recommended global git ignore

*.pem
*.key
*.p12
*.sqlite
*.db

14. Troubleshooting

Hugging Face 404 for local or default

Cause: request sent "model": "local" or "model": "default" to mlx_lm.server. The server tried to resolve it as a Hugging Face repo. 

# Fix .env:
ORCH_MODEL=$HOME/ai/models/orchestrator-qwen36-35b-a3b-heretic-bf16
DEV_MODEL=$HOME/ai/models/developer-qwen36-27b-heretic2-mixed94
REVIEW_MODEL=$HOME/ai/models/reviewer-qwen36-27b-heretic-bf16

mlx_lm.generate or mlx_lm.server not found

cd ~/ai
source .venv/bin/activate
which mlx_lm.server
uv pip show mlx-lm

# If missing:
uv pip install -U mlx-lm

Machine 1 cannot reach Machine 2

ping 10.10.10.2
ssh admin@10.10.10.2
curl http://10.10.10.2:8002/v1/models

Port already in use

lsof -nP -iTCP:8001 -sTCP:LISTEN
kill <PID>

Model outputs visible reasoning / gets cut off

  • Increase max_tokens.
  • Add system instruction: Answer directly. Do not show reasoning.
  • Use higher max tokens for Developer: 8192–16000.
  • Use lower temperature for Reviewer: 0.1–0.25.

Mac feels sluggish

  • Check memory_pressure.
  • Stop BF16 Orchestrator and switch to mixed-9bit.
  • Reduce context length if using a runtime setting that controls it.
  • Stop experimental models.

16. Instructions for Future AI Agents

Read this section before modifying the setup.

System identity

You are operating in a two-machine local AI development stack. Machine 1 is the control plane and runs the Orchestrator. Machine 2 is the worker inference server and runs Developer and Reviewer models.

Do not break these invariants

  • Do not use model=local or model=default with mlx_lm.server.
  • Use full model paths that are valid on the server machine being called.
  • Machine 1 Orchestrator endpoint is http://127.0.0.1:8001/v1.
  • Machine 2 Developer endpoint is http://10.10.10.2:8002/v1.
  • Machine 2 Reviewer endpoint is http://10.10.10.2:8003/v1.
  • Use Thunderbolt IP 10.10.10.2, not .local, for Machine 2 model traffic.
  • Do not expose model servers to the public internet.
  • Do not run destructive commands or modify unrelated directories.
  • Use worktrees for serious implementation tasks.
  • Human approval is required before merge.

When asked to implement a feature

1. Prefer frontier planning first.
2. Create plans/YYYY-MM-DD-feature-name.md.
3. Make the plan specific and executable.
4. Run local-exec-plan against the plan.
5. Inspect diff and tests.
6. Ask frontier model for final review.
7. Human merges.

When asked to troubleshoot

1. Check endpoint health:
   curl http://127.0.0.1:8001/v1/models
   curl http://10.10.10.2:8002/v1/models
   curl http://10.10.10.2:8003/v1/models

2. Check model paths in .env.

3. Check ports:
   lsof -nP -iTCP:8001 -sTCP:LISTEN
   lsof -nP -iTCP:8002 -sTCP:LISTEN
   lsof -nP -iTCP:8003 -sTCP:LISTEN

4. Check logs:
   tail -100 ~/ai/logs/orchestrator.log
   ssh admin@10.10.10.2 'tail -100 ~/ai/logs/developer.log'
   ssh admin@10.10.10.2 'tail -100 ~/ai/logs/reviewer.log'

Most likely failure modes

SymptomLikely causeFix
Hugging Face 404 for local/default Wrong model field Use full model path in .env
Cannot reach 10.10.10.2 Thunderbolt Bridge down or wrong IP Check Network settings and ping 10.10.10.2
Port 8002/8003 unavailable Machine 2 worker servers not running Run ~/ai/bin/start-worker-models.sh on Machine 2
Machine 1 slow BF16 Orchestrator memory/bandwidth pressure Switch Orchestrator to mixed-9bit fallback
Local agent modifies wrong files Plan too vague or workspace wrong Use worktree and more specific plan

Preferred final answer style for future AI agents

When helping with this setup, be concrete. Give exact commands. State which machine the command runs on. Avoid vague advice like “restart the server”; specify the port, script, path, and expected output.
