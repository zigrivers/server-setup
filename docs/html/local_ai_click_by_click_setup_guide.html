Two Mac Studio local AI · click-by-click setup

Local AI Setup Guide

$ cat ~/ai/SETUP.html — configure once, delegate forever.

A complete, standalone setup manual for the two-machine Apple Silicon AI stack: Machine 1 as the cockpit and BF16 Orchestrator, Machine 2 as the inference worker for Developer and Reviewer, connected over Thunderbolt Bridge.

Canonical procedure

This page is the canonical step-by-step. The Markdown docs under docs/ are reference material and assume you've followed these steps. Every shell command here either references a checked-in repo file or installs a checked-in repo file; nothing is hand-written into ~/ai.

Machine 2

Network

Serving

Architecture at a glance

Machine 1 / Daily Driver Machine 2 / Inference Server Claude Code / Codex / Cursor LangGraph controller MCP bridge Orchestrator BF16 127.0.0.1:8001 │ │ Thunderbolt Bridge ▼ 10.10.10.1 ━━━━━━━━━━━━━━━━━━━━━━ 10.10.10.2 Developer 8002 Reviewer 8003 Optional MTP / Security 8004

The operating model is simple: frontier models write plans and final reviews; local open-source models execute the implementation loop. The plan file is the contract.

Part 0 · macOS prep (do this first on both Macs)
P0 / Both Macs

macOS prep

Settings that the rest of this guide assumes are configured. Full detail and rationale in docs/MACOS_PREP.md; this section is the minimum required.

Set distinct hostnames

bash · on Machine 1
sudo scutil --set ComputerName  "mac-studio-1"
sudo scutil --set HostName      "mac-studio-1.local"
sudo scutil --set LocalHostName "mac-studio-1"
bash · on Machine 2
sudo scutil --set ComputerName  "mac-studio-2"
sudo scutil --set HostName      "mac-studio-2.local"
sudo scutil --set LocalHostName "mac-studio-2"

Disable sleep on Machine 2

The worker Mac must not sleep — App Nap and display sleep can suspend mlx_lm.server jobs. Machine 1 can sleep its display.

bash · on Machine 2
sudo pmset -a sleep 0 disksleep 0 displaysleep 0
sudo pmset -a powernap 0 lidwake 0 womp 1
bash · on Machine 1
sudo pmset -c sleep 0 displaysleep 20 disksleep 0
sudo pmset -c powernap 0

Disable iCloud Desktop/Documents sync

System Settings → [Apple ID] → iCloud Drive → Drive Folders… Turn off Desktop & Documents Folders on both Macs. Keep ~/ai outside any iCloud-synced folder; model weights and agent memory must never sync.

Sync time on both Macs

bash · on both Macs
sudo systemsetup -setusingnetworktime on
sudo systemsetup -getnetworktimeserver

Quick verification

bash · run on each Mac
scutil --get LocalHostName
pmset -g | grep ' sleep'
sudo systemsetup -getremotelogin     # only required on Machine 2
Pass condition
Hostnames are distinct, M2 sleep values are 0, Remote Login is On on M2.
Part I · Hardware, network, and base tools
01 / Both Macs

Thunderbolt Bridge network

Create the private direct link between the two Mac Studios. Use one real Thunderbolt 4/5 cable first; two cables do not automatically double throughput.

Connect one Thunderbolt cable

  1. Use a real Thunderbolt cable, not a generic USB-C charging cable.
  2. Connect Machine 1 to Machine 2.
  3. Open System Settings → Network on both Macs.
  4. Confirm Thunderbolt Bridge appears. If not, add it with … → Add Service → Thunderbolt Bridge.

Assign static Thunderbolt IPs

On Machine 1: System Settings → Network → Thunderbolt Bridge → Details → TCP/IP.

  • Configure IPv4: Manually
  • IP Address: 10.10.10.1
  • Subnet Mask: 255.255.255.0
  • Router: leave blank

On Machine 2, use 10.10.10.2 with the same subnet mask and blank router.

bash · from Machine 1
ping -c 3 10.10.10.2

Enable SSH on Machine 2

  1. On Machine 2: System Settings → General → Sharing.
  2. Turn on Remote Login; allow your admin user.
  3. Find the actual short username on Machine 2:
bash · on Machine 2
whoami

From Machine 1, test SSH (replace admin with the Machine 2 short username):

bash · from Machine 1
ssh admin@10.10.10.2

03 / Both Macs

Hugging Face auth

Authenticate Hugging Face so model downloads work. Use a token with read access to gated repos you intend to use.

Log in with the hf CLI

Why this is its own section

Earlier drafts buried this inside the package install. The hf CLI is now a base dependency of this repo (pulled by huggingface_hub[hf_xet]), so you only need it once you've cloned the repo and installed it. The login itself caches a token in ~/.cache/huggingface/, independent of any venv.

Run this on each Mac after you've cloned and installed the repo (Parts II–IV). When asked to add the token as a git credential, type y.

bash · on each Mac
hf auth login
hf auth whoami

14b / Machine 1

End-to-end acceptance test

Run the full frontier-to-local handoff once before considering setup complete. This exercises the MCP bridge, local-exec-plan, the worktree creation flow, and the reviewer's approval JSON — the actual user-visible outcome of all this setup.

When to run this

After Part VI is done (skills installed, MCP registered). If you do it earlier, the /delegate-local path doesn't exist yet — you can still drive the same flow via the local-exec-plan console script.

Create a disposable target repo

bash · on Machine 1
mkdir -p ~/ai/workspaces/e2e-acceptance
cd ~/ai/workspaces/e2e-acceptance
git init -q
git config user.email "local-agent@example.local"
git config user.name  "Local AI Agent"

cat > README.md <<'EOF'
# E2E acceptance target
EOF

mkdir -p plans docs/ai
cp ~/ai/local-ai-stack/docs/repo-templates/AGENTS.md      ./AGENTS.md
cp ~/ai/local-ai-stack/docs/repo-templates/CLAUDE.md      ./CLAUDE.md
cp ~/ai/local-ai-stack/docs/repo-templates/AI_WORKFLOW.md ./docs/ai/AI_WORKFLOW.md

git add . && git commit -q -m "Initial commit"

Write a tiny plan file

bash
cat > plans/$(date -u +%Y-%m-%d)-e2e-smoke.md <<'EOF'
# E2E smoke plan

## Goal
Update README.md to add a section titled "E2E acceptance" that states the
split-machine multi-agent stack ran a real plan against a real repo.

## Non-goals
Do not modify any other file. Do not add dependencies.

## Files expected to change
- README.md

## Tasks
1. Open README.md.
2. Append a section "## E2E acceptance" with one sentence noting the test
   ran successfully.

## Acceptance criteria
- README.md contains the heading "## E2E acceptance".
- git diff is limited to README.md.

## Test plan
git diff -- README.md
EOF
git add plans/*.md && git commit -q -m "Add e2e smoke plan"

Execute the plan locally

bash · on Machine 1
cd ~/ai/local-ai-stack
source .venv/bin/activate
cd ~/ai/workspaces/e2e-acceptance
local-exec-plan "$(ls -1 plans/*.md | tail -1)" --workspace .

Or from inside Claude Code (after restart with skills + MCP loaded):

text · inside Claude Code
/delegate-local execute the plan in plans/ in a separate worktree

Verify the result

bash
git worktree list
WT="$(git worktree list --porcelain | awk '/^worktree/{print $2}' | grep e2e-acceptance-local)"
echo "Worktree: $WT"
git -C "$WT" status --short
git -C "$WT" log --oneline -5
git -C "$WT" diff main -- README.md
grep "E2E acceptance" "$WT/README.md"
cat "$WT/.agent_memory.jsonl" | tail -1 | python3 -m json.tool
Pass conditions (all must hold)
  1. A new worktree exists under ~/ai/workspaces/e2e-acceptance-local-… on its own ai/local/… branch.
  2. README.md in that worktree contains the heading ## E2E acceptance.
  3. The diff is limited to README.md — no unrelated files changed.
  4. The last entry in .agent_memory.jsonl is a reviewer event with "approved": true, or a graceful stop at the max-loops cap with actionable feedback.

If any of the four fail, do not proceed to autostart. Check m1-ai-status.sh, m2-ai-status.sh, and tail ~/ai/logs/*.log — the issue is almost always an endpoint that came up sluggish or a model path that doesn't exist on the target machine.

Part VII · Autostart on boot
18 / Machine 1

Launchd autostart Machine 1

Install the Orchestrator as a per-user LaunchAgent so it starts after first login on each reboot. KeepAlive is off — if the server crashes, you'll see it in m1-ai-status.sh and can restart manually.

Install the Machine 1 LaunchAgent

bash · on Machine 1
cd ~/ai/local-ai-stack
scripts/install-launchd-machine1.sh

The script renders configs/launchd/com.localai.orchestrator.plist.template with your real $HOME, writes it to ~/Library/LaunchAgents/com.localai.orchestrator.plist, and launchctl bootstraps it. Re-run it any time you want to reload the agent.

Verify
launchctl print gui/$(id -u)/com.localai.orchestrator shows the agent loaded.
19 / Machine 2

Launchd autostart Machine 2

Install the workers LaunchAgent so Developer and Reviewer start after first login.

Install the Machine 2 LaunchAgent

bash · on Machine 2
cd ~/ai/local-ai-stack
scripts/install-launchd-machine2.sh

Same pattern: substitutes __HOME__, writes ~/Library/LaunchAgents/com.localai.workers.plist, bootstraps it. The plist invokes scripts/start-worker-models.sh, which starts both Developer (8002) and Reviewer (8003) and handles port-collision detection.

FileVault note

LaunchAgents start on first user login after reboot, not at boot time. If Machine 2 reboots while FileVault is locked, the workers won't start until someone logs in.

Uninstall (when needed)

bash · Machine 1
launchctl bootout gui/$(id -u)/com.localai.orchestrator
rm ~/Library/LaunchAgents/com.localai.orchestrator.plist
bash · Machine 2
launchctl bootout gui/$(id -u)/com.localai.workers
rm ~/Library/LaunchAgents/com.localai.workers.plist

23 / Reference

Troubleshooting

Common failures and exact fixes. Long version: docs/TROUBLESHOOTING.md.

HF 404: local/default

Your .env uses an alias. Replace with full paths on the serving machine.

Cannot reach Machine 2

ping 10.10.10.2. If it fails, check Thunderbolt Bridge IPs and cable.

mlx_lm not found

cd ~/ai/local-ai-stack && source .venv/bin/activate. Reinstall with uv pip install -e '.[mlx]'.

Output cuts off

Raise max_tokens. Orchestrator 4096–8192, Developer 8192–16000, Reviewer 4096–8192.

M1 feels slow

Switch Orchestrator to the mixed-9bit fallback. Watch memory_pressure.

MCP tool times out

Increase Codex tool_timeout_sec to 3600.

bash · quick checks
m1-ai-status.sh
ssh admin@10.10.10.2 'm2-ai-status.sh'
lsof -nP -iTCP:8001 -sTCP:LISTEN
ssh admin@10.10.10.2 'lsof -nP -iTCP:8002 -iTCP:8003 -sTCP:LISTEN'
tail -100 ~/ai/logs/orchestrator.log
ssh admin@10.10.10.2 'tail -100 ~/ai/logs/developer.log; tail -100 ~/ai/logs/reviewer.log'
24 / Reference

Security rules

Guardrails that keep the local stack useful without letting agents damage the machine. Full list: docs/SECURITY.md.

  1. Never expose mlx_lm.server, LM Studio, MTPLX, or MCP servers to the public internet.
  2. Use 127.0.0.1 for the Machine 1 Orchestrator and 10.10.10.2 for Machine 2 worker servers.
  3. Use Tailscale for remote access, not public port forwarding.
  4. Do not let local agents work outside a git worktree.
  5. Do not set ALLOW_UNSAFE_TOOLS=1 unless the workspace is disposable.
  6. Do not paste secrets into prompts.
  7. Keep .env, keys, certificates, and databases out of git.
  8. Human approval is required for dependency additions, migrations, destructive commands, and final merge.
