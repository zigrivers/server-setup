Interactive Workflow Guide

Split-Machine Development with Antigravity

A step-by-step guide for non-technical users to build features, write tests, and run local open-source models using the Antigravity CLI (agy).

01

The Local split-machine concept

The split-machine stack isolates planning and execution. A frontier model running inside the Antigravity CLI (agy) on your daily driver computer acts as the architect, while local open-source models running on an inference worker computer perform the heavy coding and testing.

1
User Requests Feature
You ask agy to build a feature or fix a bug in plain English.
▼
2
Frontier Model Creates a Plan
The high-powered frontier model writes a detailed step-by-step contract (implementation plan) and saves it to plans/.
▼
3
User Reviews and Approves
You read the plan and approve it. The stack will not start execution until you say go.
▼
4
Local Agents Execute the Plan
The local stack spins up a clean code sandbox (git worktree) and sets local models (Developer & Reviewer) to write code and verify tests.
▼
5
Review & Merge
Once the local stack finishes and the Reviewer model approves the tests, you inspect the changes and merge the worktree into main.
02

One-Time system configuration

Configure your daily driver (Machine 1) so the Antigravity CLI has access to the local stack tools and commands.

Install Global Antigravity Skills
Run the helper installer script. This copies the necessary workflow instructions so the CLI agent knows how to delegate tasks.
Terminal · Machine 1
cd ~/ai/local-ai-stack ./scripts/install-antigravity-skills.sh
Register the Model Context Protocol (MCP) Bridge
The MCP bridge connects your Antigravity agent to the local runner process. Run this script to add the configuration to your CLI automatically.
Terminal · Machine 1
./scripts/register-mcp-antigravity.sh
💡 What this did
This script modified ~/.gemini/antigravity-cli/mcp_config.json, adding the local-ai-delegate server configurations, allowed repository directories, and model endpoint URLs safely.
03

Preparing a project repository

Every project repository that you want to develop using local models needs to opt in by including local-agent instructions.

Bootstrap the Repo templates
Navigate to your target project folder (e.g. ~/ai/workspaces/my-app) and copy the instructions templates from the stack repo.
Terminal · Machine 1
cd ~/ai/workspaces/my-app STACK=~/ai/local-ai-stack mkdir -p docs/ai plans cp "$STACK/docs/repo-templates/AGENTS.md" ./AGENTS.md cp "$STACK/docs/repo-templates/CLAUDE.md" ./CLAUDE.md cp "$STACK/docs/repo-templates/AI_WORKFLOW.md" ./docs/ai/AI_WORKFLOW.md touch plans/.gitkeep git add AGENTS.md CLAUDE.md docs/ai/AI_WORKFLOW.md plans/.gitkeep git commit -m "Add hybrid AI workflow instructions"
04

Starting up the Local AI engines

Ensure your orchestration and inference servers are active before requesting work from agy.

Start the Orchestrator on Machine 1
Fire up the routing and coordination model on port 8001:
Terminal · Machine 1
start-orchestrator.sh
Verify Workers on Machine 2
If you have configured autostart daemons, the Developer and Reviewer engines will automatically be running. You can run preflight check to ensure connections are healthy:
Terminal · Machine 1
preflight.sh
💡 What this checks
The preflight checks that the Thunderbolt Bridge is connected, SSH works, and the Developer (port 8002) and Reviewer (port 8003) endpoints respond with OK.
05

Walkthrough: Build a new feature

Here is a step-by-step example of how a developer adds a database migration and a route using agy and the local split-machine models.

1. Launch the Antigravity CLI
Enter your project workspace and open agy:
Terminal · Machine 1
cd ~/ai/workspaces/my-app agy
2. Propose Feature & Generate Plan
Ask agy to create the implementation plan. Example prompt:
Antigravity CLI prompt
Create an implementation plan to add a users table database migration and route.

The CLI's frontier model will analyze the repository structures, plan the exact tasks, and write them to a file: plans/2026-05-23-add-users-db.md.

3. Delegate Execution to Local Stack
Once you approve the plan, delegate its execution to the local worker machines:
Antigravity CLI prompt
/delegate-local plans/2026-05-23-add-users-db.md
⚡ What happens under the hood
1. local-exec-plan creates an isolated git worktree sandbox.
2. The Developer model on Machine 2 writes the database migration file and the API route.
3. The Developer model writes automated tests for the route.
4. The Reviewer model on Machine 2 executes the tests and audits the code.
5. The models collaborate and fix errors until all tests pass.
4. Inspect & Merge
The local stack completes successfully and displays the changes. Exit agy, go to the generated worktree to inspect, and merge the branch when satisfied:
Terminal · Machine 1
cd ~/ai/workspaces/my-app git diff ai/local/add-users-db-... # If happy, merge the local work branch git merge ai/local/add-users-db-...
06

Troubleshooting & Triage cheat sheet

A quick-reference list of common issues when executing plans, their potential causes, and how to fix them.

Symptom Common Cause Remediation Action
local_ai_status reports FAIL on an endpoint Inference server is down or Thunderbolt bridge is disconnected. Run preflight.sh to trace the network path. Restart workers via start-worker-models.sh if needed.
run_local_plan fails with Binary not found The virtual environment has not been initialized or registered. Make sure you ran uv pip install -e '.[all]' on Machine 1, and ensure `LOCAL_EXEC_PLAN_BIN` in mcp_config.json points to your virtual env's bin path.
The agent says it has insufficient directory permissions The repository you are trying to edit is not registered in the allowed roots. Update LOCAL_AI_ALLOWED_ROOTS inside your ~/.gemini/antigravity-cli/mcp_config.json configuration to include your project root.
Local Developer fails to fix tests after 3 loops The implementation plan is too complex or ambiguous. Decline execution, break the plan down into smaller tasks, and execute again.
